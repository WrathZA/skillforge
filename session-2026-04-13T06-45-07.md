**What happened:**
1. bm-skill-recap found multiple issues (description missing: discovery loop, bm-hitl mention, 250-char overflow)
2. bm-update-a-skill Phase 0 drift-detected and offered `(s)uggest fixes`
3. I proposed description fix as item 0, then asked "what other changes?" — user said `n`
4. I interpreted `n` as "no more changes, list is complete" and jumped to bm-judge with only 1 change applied
5. User correctly pointed out: **all changes should be applied before judging**

**Root cause:** I conflated "no more changes to elicit" with "the list is complete." The recap had already identified the full set of issues — I should have proposed ALL of them as Phase 1 items upfront, let the user approve/skip each, then applied the confirmed set, then judged once.

**What correct behavior looks like:**
- Phase 0 drift check produces a set of candidate fixes
- Phase 1 elicitation presents ALL candidates (from recap + user additions) as a proposed list
- User approves the full list with `y`
- Phase 2 applies ALL confirmed changes together
- Phase 3 judges the result once
